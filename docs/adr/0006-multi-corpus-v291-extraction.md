# ADR-0006: Multi-Corpus V2.9.1 Extraction for Fidelity Validation

## Status

Accepted

## Date

2026-04-23

## Context

The V2+ FHIR Implementation Guide is meant to be a faithful, machine-readable
representation of the HL7 V2.9.1 standard. The fidelity of the FHIR resources
is the project's most consequential property — if the IG misrepresents the
underlying standard, every downstream consumer is wrong in a hard-to-detect way.

To date the project has produced V2.9.1-derived data through two extraction
pipelines, performed at different times, using different toolchains:

1. **Pandoc-derived FHIR resources.** The original FHIR StructureDefinitions
   under `input/sourceOfTruth/` (segments, messages, message structures,
   events, choreographies, complex data types) were generated by piping each
   `.docx` chapter through pandoc and then transforming the result into FHIR
   JSON. This is the body of the IG as currently published.
2. **`python-docx` re-extraction (`v291-extracted/`).** Built later by
   `tooling/scripts/extract_v291.py`, which walks the OOXML of each chapter
   directly via python-docx, parses the message-structure / segment-field /
   data-type tables, and emits a normalized JSON representation of every
   table occurrence. The corresponding comparison tooling
   (`compare_message_structures.py`, `compare_segments.py`,
   `compare_data_types.py`, `compare_v291_occurrences.py`) feeds a
   reconciliation pipeline that has so far surfaced and resolved hundreds
   of FHIR ↔ V291 disagreements.

These two corpuses have produced a substantial discrepancy report. Each fix
applied so far has either (a) corrected the FHIR resource to match V291, or
(b) corrected the V291 extraction to match a known-good FHIR value.

The remaining problem: **both pipelines share systematic blind spots**.
Specifically:

- python-docx walks tables by column index; multi-column tables with hidden
  or merged columns (e.g., QBP_Q11 in CH05) silently drop cells.
- Caption-detection in python-docx relies on heading-style heuristics; when a
  Word document reuses a heading style (e.g., the second `ACK^N02^ACK` table
  inside an `NMD_N02` clause), the caption is silently misattributed.
- pandoc table extraction has its own failure modes: misaligned headers,
  multi-paragraph cells flattened to single lines, `<br>`-as-newline being
  lost, and footnote markers leaking into cell text.

A discrepancy that exists *only* between FHIR and V291 cannot tell us which
side is wrong, only that they disagree. A discrepancy that exists *across
multiple independent extractions* gives us much stronger evidence about
which value is correct.

## Decision

Stand up two additional V2.9.1 extraction corpuses, each targeting the same
JSON schema as `v291-extracted/message-structures/*.json`, so that the
existing comparison machinery applies unchanged:

1. **LLM-mediated extraction (`v291-llm/`).** Use Claude Sonnet 4.6 with
   prompt caching against either the `.docx` chapters (text-extracted) or
   the corresponding PDFs (rendered pages). The model is asked to identify
   message-structure tables, segment-field tables, and data-type tables, and
   to emit one JSON record per occurrence in the established schema. The
   model uses *semantic* understanding of "this is a message structure
   table" rather than column-index walking, so it should be robust to the
   structural failure modes that bit python-docx (hidden columns, caption
   reuse, malformed cells).
2. **Pandoc redo (`v291-pandoc/`).** Re-run pandoc against each chapter,
   parse the resulting markdown / HTML tables, and serialize into the same
   schema. This is qualitatively different from the original pandoc → FHIR
   pipeline because the *output target* is the V291 schema, not directly
   FHIR — we get a clean, comparable third corpus rather than a derived
   FHIR artifact.

The original FHIR resources (i.e., the *implicit* pandoc corpus) remain
the published artifact. The new corpuses are *validators*, not replacements.

## Comparison Framework

With four independent representations of the same source standard, fidelity
becomes a **consensus question**:

```
                          docx (V2.9.1 source)
                                  |
              +-------------------+--------------------+--------------+
              |                   |                    |              |
        original pandoc      python-docx        LLM-mediated     pandoc redo
              |                   |                    |              |
              v                   v                    v              v
        FHIR resources    v291-extracted          v291-llm       v291-pandoc
        (input/SoT/)        (existing)             (new)            (new)
              \                   |                    |              /
               \                  v                    v             /
                +-----> 4-way comparison report (per-occurrence) <--+
```

The reporting tooling will classify every cell-level disagreement into one
of these consensus buckets:

| Bucket | Interpretation | Action |
|--------|----------------|--------|
| **4 of 4 agree** | Strong evidence the value is faithful to V2.9.1. | None — implicit confirmation. |
| **3 of 4 agree** | The dissenter is the suspect. Likely an extraction artifact. | Investigate the dissenter's pipeline; usually a fix to the extractor, not the source. |
| **2 of 2 split** (e.g., FHIR + pandoc-redo vs. python-docx + LLM) | Often signals a *transformation* difference, not a source difference (e.g., one pair preserves a "Segment" suffix, the other strips it). | Decide on the canonical representation; reconcile both pairs. |
| **All 4 disagree** | Source ambiguity or a tooling bug that's universal. | Manual review against the .docx (and likely the published PDF). |
| **N/M present** (one or more pipelines silently dropped the cell) | Often the most diagnostic case — reveals a parser blind spot. | Fix the dropping pipeline; re-extract; recompare. |

The intended deliverable is a per-structure consensus report similar in
spirit to today's `message-structure-comparison-report.html`, but with four
columns and a consensus column. Items that score "4 of 4 agree" can move
out of the manual-review queue and be considered settled. Items that score
otherwise are the target of the next reconciliation pass.

## Consequences

### Good

- Fidelity moves from a two-way disagreement (which we can only resolve by
  argument) to a four-way consensus (which we can resolve by evidence).
- Each pipeline's blind spots are exposed by the others, allowing iterative
  improvement of all three extractors over time.
- Once a value is in the "4 of 4 agree" bucket, it can be considered
  validated against the source standard with high confidence.
- The new corpuses live in their own directories (`v291-llm/`,
  `v291-pandoc/`) and never overwrite the FHIR resources. They are
  diagnostic artifacts, not part of the published IG.

### Bad

- Three corpuses to keep in sync as the source `.docx` files evolve. Future
  versions of the V2 standard will require re-running all three pipelines.
- LLM extraction has nonzero hallucination risk, so its output must always
  be cross-validated against at least one other corpus before being used to
  drive a fix.
- Token cost for the LLM pass on all 17 chapters is non-trivial (rough
  estimate: ~1M tokens for a single full pass, more if iterating on prompts).
- Adds a dependency on prompt engineering quality. Poor prompts produce
  systematically biased output that can mimic real source content closely
  enough to look like genuine evidence.

### Neutral

- Future extractions (e.g., for V2.10 when it ships) will benefit from
  having a multi-corpus methodology already in place; the cost is paid once.
- The schema discipline forced by targeting a common JSON shape may also
  pay off when other consumers want to read V291 structurally — they have a
  well-defined target to match against.

## Out of Scope

- This ADR does not commit to using the LLM extractions as a *source* for
  the FHIR resources. The FHIR resources remain the published artifact;
  multi-corpus extraction is a *validator*, not a *generator*.
- This ADR does not address vocabulary / terminology comparison (CH02C,
  Oemig's IG, UTG/THO), which has its own three-way comparison documented
  separately in `vocabulary_comparison.md`.
